# @cdt/testnet — CD lifecycle on the Cardano preview testnet

Runs the full Certificate of Deposit Token (CDT) lifecycle against the **real
Cardano preview testnet** (not the emulator) and records verifiable evidence
(tx hashes) in [TESTNET-RUN.md](./TESTNET-RUN.md).

The scripts use the on-chain validators from `onchain/plutus.json` (CIP-57
blueprint, Plutus v3) and the transaction builders from
[`@cdt/txlib`](../cdt-txlib/).

## What the lifecycle run does

1. **CD 1 — redeem at maturity.** The issuer (credit union) mints a CDT for
   the member and locks principal + full interest at the vault; the oracle
   co-signs to attest the (simulated) bank deposit. The CD matures ~12 minutes
   later (preview slot = 1 s), then the member redeems: the CDT is burned and
   the member receives principal + full interest.
2. **CD 2 — early withdrawal with penalty.** A second CD is minted with a
   1-year maturity. After ~2 minutes of accrual the member withdraws early:
   the CDT is burned, the member receives principal + accrued − penalty, and
   the remainder of the locked funds returns to the issuer.

## Prerequisites

- Node 22+ and npm.
- Build the tx library once so the `file:` dependency has its `dist/`:

  ```sh
  (cd ../cdt-txlib && npm ci)
  ```

- Install this package:

  ```sh
  npm ci
  ```

## Rerunning

```sh
npm run setup      # 1. generate (or reuse) issuer / member / oracle keys
npm run fund       # 2. faucet the issuer, distribute to member + oracle
npm run lifecycle  # 3. run both CDs end-to-end (~20 minutes)
```

All three scripts are idempotent: `setup` reuses existing keys, `fund` skips
wallets that already hold enough, and `lifecycle` mints fresh deposit ids on
every run (timestamped, so reruns never collide).

### Funding notes

`npm run fund` first tries the official preview faucet API programmatically
(`https://faucet.preview.world.dev.cardano.org/send-money/<address>?api_key=...`),
using the long-public community preview api_key as the default (the web form
is captcha-gated; the API requires a key). If the request is rejected (key
rotated, rate limit), the script prints the issuer address plus manual
instructions and polls the balance for up to 10 minutes (configurable via
`FUND_TIMEOUT_MS`):

- Faucet page: <https://docs.cardano.org/cardano-testnets/tools/faucet>
  (select **Preview Testnet**, paste the issuer address).
- One faucet grant (~10,000 tADA) is plenty; the issuer distributes working
  balances to the member and oracle itself.
- Please return leftover tADA to the faucet when done (see the faucet page).

## Provider / environment configuration

| Variable | Default | Purpose |
|----------|---------|---------|
| `KOIOS_URL` | `https://preview.koios.rest/api/v1` | Koios REST endpoint (keyless public tier) |
| `KOIOS_TOKEN` | _(unset)_ | Optional Koios bearer token for higher rate limits |
| `FAUCET_URL` | `https://faucet.preview.world.dev.cardano.org/send-money` | Preview faucet API base |
| `FAUCET_API_KEY` | community preview key | Faucet api_key query param override |
| `FUND_TIMEOUT_MS` | `600000` | How long `npm run fund` polls for manual funding |

The scripts use the keyless public Koios tier by default — no account or API
key required. Submissions and reads are wrapped in retry/backoff to absorb
public-tier rate limits and indexing lag.

## Key safety

- Keys are throwaway **preview testnet** Ed25519 keys with no real-world
  value. They are generated by `npm run setup` and stored in `.keys/`
  (`issuer.sk`, `member.sk`, `oracle.sk`, mode 0600).
- `.keys/` is **gitignored** (see [.gitignore](./.gitignore)). Never commit
  private keys or seed phrases — not even testnet ones.
- Addresses and verification key hashes are public and safe to publish; they
  appear in TESTNET-RUN.md.
- Do not reuse these keys on mainnet or send real ADA to them.

## Design notes

- The mint transaction is built directly in `src/lifecycle.ts` rather than
  with `@cdt/txlib`'s `buildMintTx`: the on-chain `cdt_mint` policy requires
  the minted CDT to sit INSIDE the vault output (with the locked lovelace
  and inline datum), while the txlib builder pays it to the owner. The txlib
  redeem / early-withdraw builders match the on-chain vault validator and
  are used as-is, as are its datum schemas and interest math.
- Confirmation is checked with a raw Koios `/tx_info` poll (`txOnChain`)
  because the installed lucid provider's `awaitTx` response schema can lag
  the Koios API. Submission and confirmation alternate in one loop so a tx
  whose validity lower bound is ahead of the node's ledger-tip slot (preview
  block gaps reach 1-2 minutes) lands once the tip advances.
